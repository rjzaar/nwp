### 2.4 Phase 3 — yq-first enforcement (commit `2ec7753`)

Replaced **11 AWK YAML parsers** with `yq` (audit estimated 3 — the
new CI gate found the rest). Net 55 lines removed.

- `pl`: `cmd_list`, `get_site_field`, `get_site_nested_field`
- `scripts/commands/stg2prod.sh`: `get_recipe_value`,
  `get_linode_config`, inline `production_config.method` reader
- `scripts/commands/prod2stg.sh`: `get_recipe_value`,
  `get_site_value`, `get_linode_server`, inline `reinstall_modules`
  reader
- `scripts/commands/doctor.sh`: `site_count` calculation (caught
  by the new CI gate at commit time — proves the gate works)

All replacements use `env()` variable interpolation rather than
shell-string interpolation, making them injection-safe regardless
of caller input validation.

**New CI job:** `lint:yq-first` in `.gitlab-ci.yml`. Refuses any
future commit that reintroduces an `awk` call reading a `.yml`
file. Non-YAML AWK uses (port extraction, du output, postfix grep,
etc.) are explicitly allowed via the regex's `(\.yml|nwp\.yml)`
anchor.

### 2.5 Phase 1 prerequisite bootstrap (commit `cd3ff04`)

Single idempotent script at `bin/setup-commit-signing.sh` that the
operator runs once to set up:

1. minisign binary (via `sudo apt-get install`)
2. SSH signing key at `~/.ssh/id_ed25519` (skips if present, never
   overwrites)
3. Minisign keypair at `keys/minisign/nwp-deploy.{key,pub}`
4. Git config: `gpg.format=ssh`, `user.signingkey`,
   `commit.gpgsign=true`, `tag.gpgsign=true` (repo-local by default;
   `--global` flag promotes)
5. `~/.ssh/allowed_signers` entry + `gpg.ssh.allowedSignersFile`
6. Smoke tests for both signing paths

One manual step printed at the end (upload SSH public key to the
operator's GitLab instance under Settings → SSH Keys).

**Threat-model note:** The minisign keypair this generates is the
runner-artifact signing key per ADR-0017 — interim software version
of the eventual Solo 2C+ hardware-rooted key. Consistent with the
key-separation model (this is the AI-accessible signing key, not
the prod-trust key).

**Status:** committed, executable, awaiting operator run.

## 3. What's in progress (blocked on operator)

### 3.1 Phase 1 — F28 CI spine

**Half done.** The bootstrap script (§2.5) is the operator's half.
The Claude half (CI wiring) is **blocked on the operator running
`./bin/setup-commit-signing.sh` first**. Without commit signing in
place, flipping the `verify-signature` job from `allow_failure: true`
to `false` would break the next push.

**When the operator unblocks**, Claude needs to:

1. **Enhance `lib/minisign.sh`** to support `NWP_MINISIGN_PASSPHRASE`
   env-var interpolation (so CI can sign non-interactively and so
   operators can avoid per-build passphrase prompts during active
   work). See §4.1 open question.
2. **Add three jobs to `.gitlab-ci.yml`** in the `deploy` stage:
   - `sign-artifact:` — invokes `lib/bundle-build.sh`
   - `upload-artifact:` — invokes `scripts/commands/publish.sh`
   - `verify-bundle:` — invokes `lib/bundle-verify.sh` (smoke
     check on the just-uploaded bundle)
3. **Flip `verify-signature` from placeholder to real** at
   `.gitlab-ci.yml:146-155` — replace the `echo` with
   `git verify-commit HEAD` and change `allow_failure: true` →
   `false`.
4. **Add a local smoke-test script** at
   `bin/smoke-test-bundle-chain.sh` that exercises
   build → sign → verify locally using `BUNDLE_NO_SIGN=1` for the
   no-key path or the real key for the full chain.
5. **Add GitLab CI Variables** (operator step) for any secrets the
   sign-artifact job needs — primarily the minisign passphrase as a
   protected/masked variable.

**Expected duration:** 4–6 h Claude-side once operator finishes
prereqs.

## 4. Open questions

These are decisions outstanding that need answers before the next
session can proceed cleanly.

### 4.1 Minisign passphrase strategy (Phase 1 follow-on)

Current state: the bootstrap script generates a passphrased
secret key. `lib/minisign.sh` doesn't yet support stdin-piped
passphrase, so every signing operation prompts interactively.

Options discussed but not decided:

| Option | Trade-off |
|---|---|
| Keep prompting | Best security; annoying for active operators (~1 prompt per deploy) |
| **Env-var pipe in `lib/minisign.sh`** (recommended) | Reads `NWP_MINISIGN_PASSPHRASE` if set; one prompt per shell session via `ssh-agent`-style use. CI needs this anyway. |
| Passphrase-less key | Most ergonomic; key file is the only protection. Reversible. |
| Solo 2C+ hardware token | End-state per ADR-0017. Not yet available. |

**Recommended next step:** Add env-var support to `lib/minisign.sh`
as part of Phase 1's CI wiring (it's needed for CI regardless).

### 4.2 Git signing config scope: repo-local vs global

The bootstrap script defaults to repo-local (`--local`) git config so
running it against the NWP repo doesn't change the operator's other
repos. `--global` is available as a flag.

**Question:** Once the operator confirms signing works, should they
promote to `--global` so all their repos get commit signing? Probably
yes for security hygiene, but it's the operator's call. The
bootstrap script prints the exact commands at the end.

### 4.3 ci-stats persistence to the `stats` branch

Phase 1.5 landed the library and the YAML anchor, but the
`stats`-branch persistence pipeline is unimplemented. Three open
items:

1. **Bootstrap the `stats` branch.** Manual `git checkout --orphan
   stats` + initial empty `.ci-stats/` + push. Documented in
   `docs/reference/ci-stats.md` §"Operations".
2. **Decide which runner has push access** to `stats`. Per the
   F36 design: one trusted runner pushes; others hand off via
   artifacts. Concrete CI plumbing TBD.
3. **Until persistence is wired**, the `.measured` anchor isn't
   wired into any actual jobs. Records would be written and lost on
   each CI run. The soak period is mostly cooling-off time for the
   library itself; real adoption waits on persistence.

### 4.4 Deployment dedupe (Phase 2) staging strategy

F36 §2 calls out that extracting `lib/deployment-core.sh` from the
six deploy scripts touches production deploy paths. The proposal
mentions "stage on the AVC stg-only deploy first" but doesn't
specify what counts as "validated enough to promote to live."

**Question for whoever starts Phase 2:** what's the smoke-test bar?
One successful stg deploy + one rollback? A specific test in
`tests/integration/`? Worth nailing down before the refactor starts.

### 4.5 What does "IMPLEMENTED" mean for multi-phase proposals?

In Phase 0 A-D1, I marked F32/F33/F34 as `IN PROGRESS` (with phase
notes) rather than `IMPLEMENTED` because they're only partially
done. The doc audit suggested `IMPLEMENTED`; I disagreed. The
broader convention question remains: do we want a status enum like
`{DRAFT, PROPOSED, ACCEPTED, IN PROGRESS, IMPLEMENTED, REJECTED,
SUPERSEDED}`? F36 §2.2 (A-D7) flags this as something to formalise
in `docs/governance/documentation-standards.md`.

**Not blocking anything**, but worth resolving before more
proposals' statuses drift.

### 4.6 Phase 0 A-C5 follow-up: `set -e` on `todo.sh`

`scripts/commands/todo.sh` already had `set -uo pipefail` before
A-C5. Adding `-e` would be a behaviour change worth testing
separately. It's in the F36 punch list but not yet scheduled.

### 4.7 Phase 0 A-C4 not yet addressed

Spotted in the architecture audit (§3.1 C4): `sync.sh:~435` uses

### 4.8 `version-changes.md` `--symlinks` cleanup (A-D3)

The doc audit found that `docs/reports/version-changes.md` still
documents the removed `--symlinks` flag as a current feature.

### 4.9 Should the `lint:yq-first` gate run on pre-commit too?

The gate runs in CI on push/MR. The audit recommended also adding
it to `.pre-commit-config.yaml` so the operator catches violations
before pushing. Easy to add but not yet done.

### 4.10 BATS run-time validation post-A-C5

The six scripts that gained `set -euo pipefail` (Phase 0 A-C5) only
had `--help` smoke-tested. The deploy scripts (`dev2stg`, `stg2prod`,
`prod2stg`) have substantial code paths that strict mode might
expose. **Recommended:** a dev-environment dry run of each before
the next production deploy.

## 6. Resume instructions

To pick this up cleanly in a new session:

1. **Read this file** for orientation.
2. **Read `docs/proposals/F36-comprehensive-codebase-improvements.md`**
   for the master plan and action-by-action status.
3. **Check `git log --oneline 9f9632d..`** for the eight F36
   commits.
4. **Decide which phase to continue**:
   - If Phase 1 prereqs are done (operator has run
     `bin/setup-commit-signing.sh` and uploaded SSH pubkey to
     GitLab), continue Phase 1 per §3.1 above.
   - Otherwise, pick from §5 (Phase 2 / 4 / 5 / 6) based on
     appetite.
5. **Don't forget the open questions in §4** — particularly
   §4.7 (`sync.sh` eval, P0 security) which is overdue.

### Workflow patterns established in these sessions

- **Multiple parallel sub-agents** for upfront audits (5 used in the
  initial F36 work). Use `Explore` subagents with scoped prompts.
- **Conventional commit prefixes:** `F36 Phase N: <subject> (action
  refs)` per the existing repo style.
- **Pre-commit gitleaks gate is operator-specific** and catches bare
  operator hostnames AND internal-domain references. The forbidden
  list lives in `.gitleaks.toml`; use role labels per F34's
  `role-vocabulary.md` and generic placeholders in all docs. Two
  leakage-gate trips happened in these sessions; both caught and
  fixed before push (one in `bin/setup-commit-signing.sh` mentioning
  the operator's GitLab domain, one in `docs/COMMAND_INVENTORY.md`
  using a bare role-eligible hostname).
- **`AskUserQuestion`** is the right tool when prerequisites or
  scope are ambiguous.
- **`WebFetch` against tool documentation** is invaluable for
  external-action input names (caught the `ddev-version` vs
  `version` bug).
